# NAS 平台差异速查

> 目标：让不同 NAS 用户先选对路径，再安装，少走弯路。

---

## 一眼判断怎么装

| 平台 | 推荐路径 | 原因 | 额外提醒 |
|---|---|---|---|
| 极空间 | 应用商店优先 | 上手最快，适合新手 | 初始化时先只配模型，飞书后补 |
| 群晖 | Docker Compose | 可控性高，社区资料多 | 注意端口、防火墙、文件夹权限 |
| 威联通 | Docker Compose | Container Station 兼容度较好 | 留意端口映射和网络模式 |
| 自建 NAS / Linux 主机 | Docker Compose 或安装脚本 | 灵活度最高 | 建议先跑通再做代理和自动化 |

## 极空间

### 推荐方式
- 优先应用商店安装
- 适合第一次接触 OpenClaw 的用户

### 优点
- 操作门槛低
- UI 路径更固定
- 不需要先理解完整 Docker 细节

### 常见问题
- Shell 入口不熟
- token 复制时被截断
- 首次启动慢，误以为坏了

### 建议
- 先完成初始化，再进 Control UI
- 模型先配 1 个，飞书后面再接

## 群晖

### 推荐方式
- Docker Compose

### 重点检查
- 共享文件夹权限
- 18789 端口占用
- DSM 防火墙策略
- `data/` 和 `config/` 的持久化

### 建议
- 如果模型访问异常，优先检查网络，再考虑 `network_mode: host`

## 威联通

### 推荐方式
- Docker Compose

### 重点检查
- Container Station 中的网络模式
- NAS 上已有服务的端口冲突
- 重启策略是否生效

### 建议
- 先从最小配置跑通，再增加飞书和自动化

## 自建 NAS / Linux 主机

### 推荐方式
- Docker Compose 或安装脚本

### 优点
- 最容易做自动化
- 最容易接入代理、备份、监控

### 建议
- 优先使用非交互安装脚本或 compose
- 把配置和数据目录纳入备份

## 统一建议

不管你是哪种 NAS，推荐顺序都一样：

1. 先让 Control UI 打开
2. 再验证模型响应
3. 再接入飞书
4. 最后做长期运行优化
